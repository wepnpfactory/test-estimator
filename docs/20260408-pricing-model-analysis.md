# 20260408 — 인쇄물 견적 가격 모델 분석 및 제안

## 0. 배경

현재 test-estimator의 가격 계산 모델은:

```
total = basePrice
  + Σ (item.addPrice × (group.perSheet ? sheets : 1) × (group.perQuantity ? qty : 1))
```

조건:
- `OptionGroup.kind`: NORMAL / SHEET_COUNT / QUANTITY / DIMENSIONS
- SHEET_COUNT 그룹의 selected item.multiplier → sheets
- QUANTITY 그룹의 selected item.multiplier → qty
- DIMENSIONS 그룹은 W×H mm 입력 → 가장 작은 fit 옵션 자동 선택 (가산만 기여)

이 모델은 "책자" 같은 단순 케이스에서는 동작하지만, **명함/소형 인쇄물/사이즈 가변 인쇄**에서는 한계가 드러난다.

---

## 1. 실제 인쇄물 견적의 가격 구성 (도메인 분석)

전형적인 인쇄물(책자/명함/포스터/스티커) 가격은 다음 요소들의 합으로 결정된다.

### 1.1 종이비
- 공식: `종이 단가(원/㎡ 또는 원/장) × 사용 장수 × 부수`
- 변수:
  - **종이 종류**: 스노우, 모조, 아트, 랑데뷰 … → 단가/㎡
  - **평량(g/m²)**: 80g/100g/250g/300g … → 같은 종이여도 평량별 단가
  - **장수**: 명함은 보통 1장, 책자는 페이지수/2 (양면 기준)
  - **부수**: 주문 수량
  - **사이즈**: 면적이 클수록 한 장당 종이 사용량 증가
- 참고: 실제로는 "원지 한 장에 몇 컷이 나오느냐(컷수)" 로 계산. 작은 명함은 대지 1장에서 수십 장 컷팅. 우리는 단순화해서 면적 비례로 처리.

### 1.2 인쇄비
- 공식: `(판비 + 인쇄단가 × 색도 × 면 × 장수) × 부수`
- 변수:
  - **인쇄기**: 윤전(대량 저단가) / 옵셋(중량 균형) / 인디고(고품질) / 토너(소량)
  - **색도**: 1도(흑백) / 4도(컬러). 5도 이상(별색) 도 있음
  - **인쇄면**: 단면 / 양면. 양면 = 인쇄비 × 2
  - **판비**: 옵셋·인디고에서 발생. 토너·윤전은 거의 0
- **인쇄기별 가격 구조 차이**:
  - **토너**: 판비 0, 장당 단가 일정. 부수가 많을수록 비싸짐
  - **인디고**: 판비 약간, 장당 단가 중간. 100~500부 sweet spot
  - **옵셋**: 판비 큼, 장당 단가 매우 낮음. 1000부 이상에서 유리
  - **윤전**: 판비 매우 큼, 장당 단가 극저. 10000부 이상

### 1.3 후가공비
- 코팅 / 박 / 형압 / 에폭시 / 귀도리 / 타공 등
- 보통 **부수당** 단가 (장당이 아님)
- 박/형압은 면적·디자인 복잡도 영향 (MVP는 무시)

### 1.4 제본비
- 책자류만 해당
- 공식: `제본 단가 × 부수` (페이지수에 따라 단가 변동)
- 무선/중철/스프링/하드커버 별 단가 다름

### 1.5 일반경비 / 배송 / 부가세
- 고정 또는 구간별
- 부가세는 소계 × 10%

## 2. 현재 모델의 한계

### 한계 ①: 다축 의존성 표현 불가
**예**: 토너 + 컬러 + 양면 조합 → 토너의 컬러 양면 단가 적용 필요. 현재는 각 옵션의 addPrice가 독립적으로 합산되므로 "토너일 때만 컬러비가 더 비쌈" 같은 조합 효과를 표현 못함.

**근거**:
- 토너 단면 흑백: 50원/장
- 토너 양면 컬러: 200원/장 (4배)
- 인디고 양면 컬러: 80원/장
- 옵셋 양면 컬러: 30원/장 (옵셋 1000부 기준)

→ "인쇄기 × 색도 × 인쇄면" 3축 매트릭스 9개 셀이 필요한데, 현재 구조로는 9개 옵션을 따로 만들거나 평균값으로 근사할 수밖에 없음.

### 한계 ②: 직접 입력 부재
SHEET_COUNT / QUANTITY 그룹은 미리 정의된 옵션들 중 선택만 가능. 실제 인쇄 주문은 "100장", "323부" 같은 임의 수량을 받아야 함.

### 한계 ③: 사이즈 면적이 가격에 반영되지 않음
DIMENSIONS 그룹은 가장 작은 fit 옵션의 addPrice만 기여. 명함 55×85mm와 A4 210×297mm는 면적이 13배 차이지만 단순 가산이라 **종이비/인쇄비 비례가 안 됨**.

### 한계 ④: 부수 구간별 단가 변화 미지원
인쇄기마다 sweet spot 부수가 다른데, 단순 곱셈은 "1부 단가 × 부수"라 항상 선형. 실제는 1000부 사면 단위가격 1/3로 떨어지는 stepwise 가격을 표현해야.

### 한계 ⑤: 판비 같은 "고정 + 부수당" 분리 안 됨
판비 100,000원은 **부수와 무관**. 1부면 100,000원 부담, 1000부면 100원/부 분담. 이 구조를 표현하려면 "부수당 곱셈 안 함 + 일반 가산" + "부수당 곱셈" 두 항목으로 분리해야 하는데 현재 모델로도 가능. 다만 관리자가 직관적으로 입력하기 어려움.

## 3. 모델 선택지

### 옵션 A — 현 모델 유지 + 직접 입력 + 면적 인식 추가
- **변경**: SHEET_COUNT/QUANTITY 직접 입력, DIMENSIONS 면적을 가격 곱셈에 활용
- **장점**: 가장 작은 변경. UI 단순.
- **단점**: 다축 의존성(토너+컬러+양면) 표현 불가. 옵션 조합 폭발 회피하려면 평균 단가로 근사 → 부정확.
- **적합**: 단일 인쇄기·고정 색도 운영. 다양한 인쇄기 비교 필요 없는 단순 카탈로그.

### 옵션 B — PriceRule 활성화 (룰 매트릭스)
- **변경**: 기존 PriceRule 모델에 평가 엔진 + 입력 UI 구현
- **룰 형식**:
  ```
  {
    "name": "토너 컬러 양면 인쇄비",
    "match": [
      { "groupValue": "press", "itemValue": "toner" },
      { "groupValue": "ink",   "itemValue": "color_4d" },
      { "groupValue": "side",  "itemValue": "double" }
    ],
    "action": "ADD_RATE",
    "amount": 200,
    "perSheet": true,
    "perQuantity": true,
    "perArea": false
  }
  ```
- **평가**: 모든 매칭 룰을 합산. 옵션의 자체 addPrice 와는 별개로 추가됨.
- **장점**: 다축 의존성 정확히 표현. PriceRule이 늘어나도 옵션 자체 모델은 단순 유지.
- **단점**: 룰 갯수 관리. 복잡한 도메인은 룰 폭발 (예: 4축 5×5×5×3 = 375 룰. 실제는 의미있는 셀만 입력하니 30~50개 정도).
- **적합**: 다양한 인쇄기·재질·색도를 비교 견적하는 본격 인쇄소.

### 옵션 C — JS 표현식 평가기 (관리자가 직접 수식 작성)
- **변경**: PriceRule 대신 sandboxed JS 함수 저장 → 평가 시 실행
- **장점**: 가장 유연. 복잡한 곱셈/구간 단가 표현 가능
- **단점**: 보안(VM 격리), 디버깅 어려움, 관리자가 코드 작성 가능해야

### 옵션 D — 외부 견적 API 위임
- **변경**: 가격 계산을 우리 시스템이 안 함. 견적 시 인쇄소 자체 견적 API에 옵션 셀렉션을 그대로 전달.
- **장점**: 정확. 우리 책임은 옵션 UI + Cafe24 동적 상품 등록만.
- **단점**: 인쇄소가 자체 견적 시스템을 갖추고 있어야 함.

## 4. 추천 — 옵션 B (PriceRule) + 옵션 A 흡수

### 4.1 전체 구조
```
total
  = basePrice
  + Σ option.addPrice × (group.perSheet ? sheets : 1)
                      × (group.perQuantity ? qty : 1)
                      × (group.perArea ? areaMm2/baseArea : 1)   ← 신규
  + Σ rule.amount × matched(rule, selections)
                  × (rule.perSheet ? sheets : 1)
                  × (rule.perQuantity ? qty : 1)
                  × (rule.perArea ? areaMm2/baseArea : 1)
```

### 4.2 변경사항

#### 스키마 추가
- `OptionGroup.allowDirectInput Boolean` — SHEET_COUNT/QUANTITY/DIMENSIONS 그룹의 직접 입력 허용
- `OptionGroup.minDirect Int?`, `maxDirect Int?` — 직접 입력 범위 (장수/부수/mm)
- `OptionGroup.perArea Boolean` — 이 그룹의 가격을 (선택된 사이즈 면적 / baseArea)에 곱함
- `Product.baseAreaMm2 Int @default(89250)` — 기본 면적 (A4 = 210×297×... 또는 명함 55×85×... 운영자가 설정)
- `PriceRule.match Json` — 조건 [{groupValue, itemValue}, ...] AND
- `PriceRule.action Enum` — ADD_FIXED / ADD_RATE
- `PriceRule.perSheet Boolean`
- `PriceRule.perQuantity Boolean`
- `PriceRule.perArea Boolean`

#### 평가 엔진
1. selections에서 면적 산출: DIMENSIONS 그룹이 있으면 입력값 W×H 또는 선택된 옵션의 widthMm×heightMm
2. selections에서 sheets/qty 산출: SHEET_COUNT/QUANTITY 그룹의 multiplier 또는 직접 입력값
3. 옵션별 contribution 합산 (perSheet/perQuantity/perArea 곱셈 포함)
4. PriceRule 매칭 + 추가 contribution 합산
5. 최종 finalPrice 반환

#### Admin UI
- 그룹 카드 고급 설정에 `allowDirectInput`, `min/max`, `perArea` 추가
- 별도 섹션 "가격 규칙" — PriceRule 빌더 (조건 행 추가 + action 선택 + 곱셈 플래그)

#### embed.js
- SHEET_COUNT/QUANTITY 그룹이 `allowDirectInput=true`면 select 대신 number input
- DIMENSIONS 그룹은 이미 W/H 입력 → 면적 자동 계산해 quote 호출
- 표시 로직 거의 그대로

### 4.3 PriceRule 입력 UI 폼 디자인
```
[+ 가격 규칙 추가]

규칙 1:  토너 컬러 양면 인쇄비
조건 (모두 만족):
  [인쇄기]   = [토너          ]
  [색도]     = [컬러 (4도)    ]
  [인쇄면]   = [양면          ]  [×]
  [+ 조건]
계산: ( + 200 ) × ☑ 장당 ☑ 부당 ☐ 면적당
```

## 5. 명함 케이스 적용 시뮬레이션

명함 100장, 토너인쇄, 양면, 스노우 300g, 컬러, 사이즈 55×85mm

### 옵션 그룹
| 그룹 | kind | perSheet | perQuantity | perArea | 비고 |
|---|---|---|---|---|---|
| 사이즈 | DIMENSIONS | - | - | - | 면적 산출용 |
| 수량 | QUANTITY | - | - | - | 직접 입력 허용 |
| 인쇄기 | NORMAL | - | - | - | 인쇄기별 식별만 |
| 양면/단면 | NORMAL | ☑ | ☐ | ☐ | 양면일 때 인쇄비 ×2 |
| 용지 | NORMAL | ☑ | ☐ | ☑ | 종이값 (장당, 면적 비례) |
| 평량 | NORMAL | ☑ | ☐ | ☑ | 평량 가산 (장당, 면적 비례) |
| 인쇄(색도) | NORMAL | ☑ | ☐ | ☐ | 컬러/흑백 단가 |
| 후가공 | NORMAL | ☐ | ☑ | ☐ | 부수당 |

### 가격 규칙 (PriceRule)
| 규칙 | 조건 | amount | 곱셈 |
|---|---|---|---|
| 토너 컬러 양면 인쇄비 | 인쇄기=토너 ∧ 색도=컬러 ∧ 양면=양면 | 200 | ×장 ×부 |
| 토너 흑백 양면 인쇄비 | 인쇄기=토너 ∧ 색도=흑백 ∧ 양면=양면 | 50 | ×장 ×부 |
| 인디고 컬러 인쇄비 | 인쇄기=인디고 ∧ 색도=컬러 | 80 | ×장 ×부 |
| 옵셋 판비 | 인쇄기=옵셋 | 100000 | (고정) |

### 계산 시뮬레이션 (명함 100장, 토너 컬러 양면, 스노우 300g, 55×85mm)
- basePrice = 5,000
- area ratio = (55×85) / (210×297) = 4675/62370 = 0.075
- 사이즈 (DIMENSIONS): 0원 (면적만 산출)
- 수량 (직접입력): 100장
- 인쇄기 토너: addPrice 0
- 양면: addPrice 0 (perSheet, area 영향 없음)
- 용지 스노우: 50원 × 100장 × 0.075 = 375
- 평량 300g: 30원 × 100장 × 0.075 = 225
- 인쇄 컬러: 0원 (룰에서 처리)
- 룰 "토너 컬러 양면 인쇄비": 200원 × 100장 × 1부 = 20,000

→ total = 5000 + 375 + 225 + 20000 = **25,600원**

(yopp 실제: 토너 100장 명함 양면 컬러 ≈ 30,000원 수준 — 근사 OK)

## 6. 결정 필요

### 6.1 어떤 옵션으로 갈 것인가?
- **A (현 모델 + 직접 입력)** — 단순. yopp만큼 정확한 견적은 못 줌.
- **B (PriceRule + 옵션 A 흡수, 추천)** — 본격적 인쇄 견적 가능. 작업량 큼 (스키마 + 평가기 + 룰 빌더 UI).
- **C (JS 표현식)** — 너무 자유로움. 관리자 학습 곡선 큼.
- **D (외부 견적 API)** — 인쇄소가 견적 시스템 보유 시.

### 6.2 사이즈 처리
- 면적 비례 곱셈을 디폴트로 둘지, 옵션별 perArea 토글로 둘지
- baseArea 를 어떻게 결정할지 (가장 큰 사이즈? 자주 쓰는 표준?)

### 6.3 수량 직접 입력
- 최소/최대 범위
- 구간별 단가 (1~50장, 51~100장, …) 표현 — 룰로?

### 6.4 인쇄기 의존성
- 인쇄기 옵션 자체에 가산 vs 룰로 처리
- 룰로 가는 게 정확하지만 룰 갯수 증가

### 6.5 PriceRule UI 입력 형태
- 폼 빌더 vs JSON 직접 입력
- 1차는 JSON, 2차는 폼?

## 6.6 레거시 상품 48종 분석으로 확인된 추가 요구사항

`docs/20260408-legacy-product-options.md` 를 참고하면 다음이 명확해진다:

### 그룹 카디널리티 제약
- **명함류 (041~048)**: paper_size_type 옵션이 90×50, 85×55 같은 **명함 전용 프리셋** + **직접입력**
- **포스터류 (031~037)**: B2/B3/B4/A2/A3/국반절 + 직접입력. 인쇄기에 따라 사용 가능 사이즈 제한
- **책자 양장 (003·004)**: 표지+내지+면지+후가공+제본+배송까지 풀스펙 — 약 **20개 이상의 옵션 그룹**

### 다축 매트릭스가 필수인 영역
- **인쇄기 × 사이즈**: 토너는 A3/B4까지, 인디고는 B2까지, 옵셋은 별도 — 인쇄기 선택 시 사이즈 옵션이 동적으로 제한
- **종이 등급 × 종이 이름**: paperGrade=일반/고급 으로 cover_paper_name 옵션이 달라짐
- **인쇄기 × 색도 × 면**: 단가가 모든 조합마다 다름
- **표지 종이 × 평량**: 종이마다 사용 가능한 평량 다름 (스노우 250g/300g, 랑데뷰 240g 만, 등)

### 조건부 노출이 자주 등장
- `cover_type=하드` 일 때만 hardcover_type / headband_color / tailband_* 노출
- `binding_type=무선날개` 일 때만 cover_left_flap_width_mm / cover_right_flap_width_mm 입력
- `binding_type=트윈링/코일링` 일 때만 ring_color 노출
- `addtype_flyleaf=on` 일 때만 면지 종이/평량/매수 노출

### 직접 입력이 표준
- 사이즈: 모든 인쇄물이 "직접입력" 옵션 보유 (가로 mm × 세로 mm)
- 수량: 모든 인쇄물에서 자유 입력
- 페이지수: 책자류에서 짝수만 자유 입력 (최대 800), 시험지(011)는 enumeration

### 결론 보강
- **B (PriceRule)** 가 사실상 강제. 옵션 단가 가산만으로는 토너/인디고/옵셋 단가 차이를 표현 불가.
- **showWhen 그룹 노출**은 이미 우리 모델에 있음. 필드별 노출(cover_type별 hardcover_type) 도 같은 메커니즘으로 가능.
- **그룹 직접 입력 + 그룹 의존 옵션 셋** 도 필수. 예: paperGrade 라는 그룹의 선택값에 따라 cover_paper_name 그룹의 enabled 옵션이 달라져야 함. → 옵션 아이템 단위 showWhen 도 필요할 수 있음 (현재는 그룹 단위만).

### 추가 결정 사항
- **OptionItem 단위 showWhen** 도입할지? (그룹 단위만으로는 "스노우 종이일 때 250g/300g 만" 같은 표현 어려움)
- 또는 **그룹 직접 입력 + 그룹 간 enabled 동기화** 메커니즘으로 처리할지
